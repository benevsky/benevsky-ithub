01 Главный вопрос архитектуры

А зачем вообще две CRM?

В исходной схеме v1 заложены Bitrix24 (Deal Pipeline, 9 стадий) и amoCRM (Investor Pipeline, 6 стадий) с двунаправленной синхронизацией. Прежде чем писать код — нужно ответить, что именно эти системы делают, чего не сделает обычная база данных.

Critical question
«Зачем amoCRM и Bitrix24, если фактически они работают как дорогая база данных с готовым канбаном?»
В архитектуре v1 вокруг обеих CRM уже навешана PostgreSQL, FastAPI, Celery, Redis, Scoring Engine и кастомные React-дашборды. Сами CRM используются для трёх вещей: хранения сущностей, отображения канбана и отправки вебхуков. Всё это решается одной БД и React-приложением — без лицензий, без rate limits, без мостов синхронизации.
v1 · Архитектура с двумя CRM
Bitrix24 + amoCRM + Postgres
3 источника состояния, 2 sync-моста
  • Двойная лицензия — оплата двух SaaS ежемесячно
  • Двунаправленная синхронизация Bitrix ↔ amoCRM — классический источник багов и расхождений данных
  • Rate limits на API обеих платформ — узкое горлышко для AI-агента
  • Кастомные поля как UF_* — методология AP (8 блоков скоринга, Level 1–4) натягивается поверх чужой схемы
  • Vendor lock-in на две платформы сразу — миграция в будущем будет в 2× больнее
  • Latency для AI-агента: каждый tool-call — это REST-хоп (~150–300мс) вместо запроса к БД (~5мс)
  • Зависимость от маркетплейсов Bitrix/amo, их апдейтов и breaking changes в API
v2 · Унифицированный стек
PostgreSQL + Custom UI
1 источник правды, 0 sync-мостов
  • Один источник правды — PostgreSQL на PlanetScale, нет рассинхронов by design
  • Кастомный UI на shadcn/ui (Radix) — production-grade компоненты, accessible, быстрая разработка
  • Доменная модель first-class — Deal, Investor, Scoring, Level — нормальные таблицы с FK, а не UF-поля
  • AI-агент работает напрямую с БД — миллисекунды вместо сотен мс, нет rate limits
  • Свобода UX — интерфейс под методологию VC/PE, а не общий канбан «для всего»
  • Независимость от чужих маркетплейсов и их breaking changes
  • Time-to-market сравним с интеграцией двух CRM — кастомный канбан на dnd-kit + shadcn делается за 2–3 недели
02 Технологический стек v2.0

03 Карта замен v1 → v2

Что и почему меняем.

Каждая позиция архитектуры v1 пересмотрена. Для каждой замены — конкретный технический аргумент. Это не «модно», это «работает быстрее и проще».

Слой Было (v1) Стало (v2) Почему
CRM / Источник правды Bitrix24 + amoCRM PostgreSQL (PlanetScale) Один источник правды, нет sync-боли, доменная модель в нормальной схеме, нет vendor lock-in на маркетплейсы.
UI / Frontend Стандартные интерфейсы CRM + кастомные дашборды Next.js + shadcn/ui Production-grade компоненты на Radix (a11y из коробки), полная свобода UX под методологию AP, единый код для портала и админки.
Backend / API FastAPI + Node.js Gateway + Kong Next.js API Routes + Cloudflare Workers Один framework на фронте и бэке, edge-deploy для low latency, серверless = нет инфраструктурного DevOps. Тяжёлые задачи (очереди, парсинг) — на Workers.
Очереди / Background jobs Celery + Redis Cloudflare Queues + Workers Serverless, нет сервера для обслуживания, distributed network = низкая latency глобально, встроенная DDoS-защита.
SLA-таймеры / Cron Celery Beat Vercel Cron + Cloudflare Cron Triggers Декларативное расписание, ноль инфраструктуры, мониторинг из коробки.
AI-агент «Игорь» LangChain + кастомная память Claude Agent SDK + Daytona + Coregit Агент SDK — production-ready под Claude, без LangChain-абстракций. Daytona — изолированные сэндбоксы для безопасного выполнения. Coregit хранит state и версии документов.
Latency-критичные AI-задачи GPT-4o на всё Mercury-2 dLLM (для быстрых) + Claude Sonnet (для сложных) Diffusion LLM Mercury-2 даёт frontier-качество на простых задачах за миллисекунды. Экономия на токенах + время отклика.
Аутентификация Кастомная (JWT RS256 + OTP + MFA) Clerk Кастомная auth — это всегда дыры безопасности. Clerk даёт OTP, MFA, OAuth, RBAC, session management — всё из коробки и надёжно.
Email SendGrid Resend Современный API, React Email для шаблонов, простая настройка домена, у нас уже есть боевой опыт.
File Storage S3 Cloudflare R2 + Coregit R2 — zero egress fees (важно для тизеров и DD-документов с массовой выдачей). Coregit — для агентских артефактов с версионированием.
Audit Log Отдельный сервис + ELK PostgreSQL append-only таблица Меньше движущихся частей, прямой SQL-доступ для AI-агента и аналитики, проще backup и compliance.
Дашборды React + Recharts (отдельный модуль) Next.js + shadcn/ui charts Единый компонент-стек с остальным приложением, готовые chart-компоненты Recharts/Tremor под shadcn.
04 AI-агент «Игорь» v2.0

Без LangChain. Claude Agent SDK + sandbox + версионируемая память.

В v1 агент построен на LangChain поверх REST-вызовов в Bitrix. В v2 — нативный Claude Agent SDK, прямой доступ к БД, изолированные sandboxes для безопасного выполнения и Coregit как persistent layer для state и документов.

05 Что мы получаем по итогу

Меньше системных частей — больше скорости и надёжности.

Оптимизация архитектуры — не упражнение в минимализме, это прямое влияние на time-to-market, операционные расходы и debug-нагрузку команды на длинной дистанции.

Меньше систем
−5
Минус Bitrix24, amoCRM, Kong Gateway, Celery+Redis, отдельный logging-стек. Итого 5 систем долой из эксплуатации.
Меньше sync-мостов
−3
Bitrix↔amo, Bitrix↔Postgres, amo↔Postgres — все три моста уходят. Один источник правды по умолчанию.
Time-to-market
−30%
Единый стек Next.js + serverless = меньше DevOps, готовые компоненты shadcn = быстрее UI, Clerk = ноль времени на auth.
Latency AI-агента
~10×
Прямой запрос к БД (5–10мс) вместо REST в Bitrix (150–300мс). На длинных reasoning-цепочках разница в секунды.
Лицензий в месяц
−2
Bitrix24 и amoCRM выходят из ежемесячных расходов. Чистая экономия операционных затрат.
Vendor lock-in
0
Ни одной зависимости от закрытых маркетплейсов. Весь stack — open standards (Postgres, S3-compat, OAuth) и production-grade managed-сервисы.
Хорошая архитектура измеряется не количеством сервисов на схеме, а количеством багов, которые туда не попадут на проде.
